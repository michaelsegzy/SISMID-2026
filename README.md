# SISMID 2026: A Human-AI Teaming Approach to Infectious Disease Modeling

**Building an Influenza Forecasting Pipeline · July 22 through July 24, 2026**

## 1. Overview

Every week during influenza season, public health decision-makers need to anticipate what is coming: how many people will be hospitalized next week, when the peak will hit, and how high it will go. In this lab you work as a human-AI team to build a national influenza hospitalization forecasting system from real CDC data.

You define the vision and the rules. AI helps build and implement the solution. Think of AI as a very capable intern: fast and helpful, but it still needs direction. The clearer the guidance, the better the results. At the end of the day, you are still the expert making the decisions.

The heart of the lab is a single document, `rules.md`, that you author in plain language. It is the blueprint that describes what the pipeline should do. Across the activities it grows from a data-cleaning spec into a full forecasting and evaluation pipeline and a FluSight-ready model. This is an AI course, not an intro to GitHub or R; those tools support the real work, which is your modeling.

## 2. End products

By the end of the lab you will have a complete, reproducible pipeline and the model it produces. Concretely:

| Product | What it is |
|---|---|
| `rules.md` | Your human-authored specification. Describes **what** the pipeline should do, in plain language. You edit this to change the output. |
| `agents.md` | Generated by AI from your rules. Describes **how** to execute the pipeline. The intermediary between you and the code. |
| Pipeline scripts | The R scripts the agent writes from `agents.md`: cleaning, exploration, forecasting, evaluation, and improvements. |
| Forecast outputs | Cleaned data, figures, a FluSight-format forecast file, and scored evaluation tables. |
| FluSight submission files | `YY-MM-DD_NAME_MODEL.csv`, your final forecasts for the course mini-challenge. |

## 3. General lab structure

Every activity runs through the same human-AI iteration loop:

1. **Edit the rules.** Open `rules.md` and write or review the rules for that phase.
2. **Generate `agents.md`.** Prompt Copilot to build an agent from your rules document.
3. **Generate the R script.** Use the agent to write the script for this phase.
4. **Run and evaluate.** Execute the script in VS Code and check the output.
5. **Update and regenerate.** Refine the rules, regenerate `agents.md`, regenerate the script.
6. **Push to GitHub.** Automated checks run on your submission and give you feedback.

You repeat this cycle for each phase. Push early and often: every push is a checkpoint, and the agent keeps a record of all your changes.

**Vague rules produce vague output.** Most of your effort goes into `rules.md`, not into typing R. Two habits make the difference. First, be precise about every detail: input paths, column names and types, plot axes and titles, file names, and output folders. Second, write validations into your rules. Diagnostics are how you confirm the agent interpreted your intent, for example printing the `(p, d, q)` order `auto.arima()` selected, confirming `target_end_date - reference_date` equals exactly 7 days, or reporting how many forecast rows were dropped for lack of an observed target.

Two icons appear throughout the slide decks. The pie-chart icon marks a slide listing the goals and expected outputs for an activity; everything on those slides gets uploaded to GitHub for review. The **YOUR TURN** stamp marks a hands-on task.

A few definitions recur across activities and are worth pasting straight into your rules when they come up.

**Influenza season.** MMWR week 40 (roughly early October) through week 20 (roughly mid-May) of the following year. A season spans two calendar years and is named for both, for example 2025-26. Season start is the first calendar date in the first year with epiweek 40; season end is the earliest date in the second year with epiweek 20. Account for leap years. When computing `season_start_year`, use `year(week)` as the base, not the `MMWRyear` from `MMWRweek()`, so weeks crossing the calendar-year boundary are not mis-attributed.

**FluSight format.** One row per quantile, with columns `reference_date`, `target` (`wk inc flu hosp`), `horizon`, `target_end_date`, `location` (`US`), `output_type` (`quantile`), `output_type_id` (the quantile), and `value` (a non-negative integer). Dates are ISO `YYYY-MM-DD` and land on a Saturday. `0.5` is the median point forecast; `0.025` and `0.975` are the bounds of the 95% prediction interval.

**A few gotchas.** At horizon 1, `pr$se` already carries the ARIMA innovation variance, so use `sd_pred <- as.numeric(pr$se)` directly rather than adding a residual-variance term. `scoringutils` returns `ae_median` per forecast; MAE only appears after averaging across reference dates. 95% coverage is not computed by default and must be added explicitly. Set `forecast_unit` explicitly in `as_forecast_quantile()`, or forecasts get silently mis-grouped.

## 5. GitHub structure & where to start

You fork this repository, work locally in VS Code, and push your results back. All student work lives under `Student_Output/`, in a folder named for you:

```
Student_Output/
└── YOUR_NAME/
    ├── rules.md
    ├── agents.md
    └── output/
        ├── scripts/      01_cleaning.R · 02_data_explore.R · 03_forecast.R
        │                 04_evaluation.R · 05_improvements.R
        ├── data/         01_cleaning/cleaned_flu_admissions.csv
        │                 02_data_explore/peak_description.csv
        │                 03_forecast/flusight_forecasts.csv
        │                 04_evaluation/forecast_scores.csv (+ _by_horizon)
        └── figures/      01_cleaning/epicurve_us_flu_admissions.png
                          02_data_explore/national_trend.png (+ seasonal_comparison)
                          03_forecast/forecast_vs_observed.png
```

Use your name exactly the same way every time you push: the automated checks find your work by folder name. If a folder path does not yet exist, write a rule telling the agent to create it before saving, and keep that rule for the rest of the lab.

You will use three GitHub tabs. **Code** shows every file, including the shared data folder and all student folders. **Pull Requests** is where your submissions land and where you see the agent's feedback and whether you passed. **Actions** is where the automated checks run; click into a run to see exactly what was checked and why.

**Where to start:**

1. **Fork this repository.** Click `Fork`, confirm your account is the owner, and click `Create fork`. You now have a personal copy you can edit freely.
2. **Clone your fork in VS Code.** Copy the HTTPS URL from the green `Code` button. In VS Code press `Ctrl + Shift + P`, run `Git: Clone`, paste the URL, choose a folder, and click `Open`.
3. **Install the R extension.** Open Extensions (`Ctrl + Shift + X`), search `REditorSupport`, and Install. You need [R](https://cran.r-project.org) installed locally first.
4. **Connect Copilot.** Click the Copilot icon at the bottom right of VS Code, choose `Continue with GitHub`, and authorize.
5. **Check your data.** Confirm the project contains `data/Weekly Hospital Respiratory Data (HRD) Metrics by Jurisdiction.csv`.
6. **Start Activity 01.** Open `rules.md` and read the Data Cleaning Rules.

To push your work, open Source Control (`Ctrl + Shift + G`), stage your files, write a commit message such as `activity_1_upload`, click Commit, then Sync Changes. If you cannot use Git locally, upload files directly through your fork on GitHub instead. If `git --version` returns nothing, see the Downloading Git helper deck before continuing.

If something breaks, bring it to the room. If something sparks your curiosity or you want to go further, our door is always open.
